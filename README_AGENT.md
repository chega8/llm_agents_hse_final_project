# Агент поддержки кредитования МСБ (LangGraph + GigaChat)

автоматизированный Помощник для обращений клиентов МСБ по
кредитованию. Подключает **RAG** по нормативке, **инструменты** к БД клиентов, корректно
**эскалирует** (продажи/негатив/безопасность) и **отказывает** в краевых случаях.

## Архитектура

```
src/credit_agent/
  config.py    # токен из ./env, фабрики GigaChat LLM и Embeddings, пути, TODAY
  prompts.py   # системный промпт по регламенту РП-ОБ-005 + контекст сессии
  rag.py       # чанкинг доков по заголовкам (с иерархией), GigaChat embeddings, FAISS
  tools.py     # read-only инструменты к clients.sqlite + расчёт ДП + escalate()
  agent.py     # сборка LangGraph create_react_agent для сессии (client_id/channel)
build_index.py # построение FAISS-индекса (запустить один раз)
chat.py        # интерактивный CLI-чат
eval.py        # E2E-оценка по qa.jsonl с метриками по категориям
```

### Ключевые решения
- **RAG**: документы режутся по заголовкам markdown с сохранением *пути* родительских
  заголовков в каждом чанке (дочерняя «Условия» несёт название продукта) — это резко
  улучшает семантический поиск. Плюс синтетический обзорный чанк со списком всех 5 продуктов.
  Каждый чанк помнит `source` и номер пункта (`2.1.5`) для ссылок на нормативку.
- **Безопасность данных**: клиентские инструменты замкнуты на `client_id` сессии и НЕ
  принимают id от модели — доступ к данным третьих лиц закрыт на уровне кода (разд. 7 РП-ОБ-005).
- **Эскалация** — инструмент `escalate(trigger, summary)` с триггерами `sales|negative|security`;
  даёт машинно-проверяемый сигнал для оценки.
- **Идентификация**: анонимные каналы (`chat_site`/`contact_center`) → только общая инфа;
  авторизованные (`chat_intern`/`mobile`) → доступ к собственным договорам/заявкам.

## Установка

```bash
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
```

Токен GigaChat читается из файла `./env` (`GIGACHAT_ACCESS_TOKEN=...`) или из переменной
окружения. Модель — `GigaChat-2`, `temperature=0.1`.

## Запуск

```bash
# 1. Построить индекс RAG (один раз; пересобрать при изменении документов)
.venv/bin/python build_index.py

# 2. Чат
.venv/bin/python chat.py                                   # анонимная сессия
.venv/bin/python chat.py --client C-000001 --channel chat_intern   # авторизованная

# 3. E2E-оценка
.venv/bin/python eval.py --limit-per-category 2            # быстрый срез по всем категориям
.venv/bin/python eval.py --categories escalation_sales,edge_manipulation --limit-per-category 3
.venv/bin/python eval.py --limit-per-category 99           # фактически полный прогон qa.jsonl
```

## Метрики оценки (eval.py)

Считаются **по категориям отдельно** (как рекомендует README кейса):
- **judge_pass** — LLM-судья (GigaChat) сверяет ответ с `expected_behavior` по сути.
- **escalation** — детерминированная проверка триггера: сработала ли эскалация там, где
  ожидается (`escalation_sales`→`sales`, `escalation_negative`→`negative`), и НЕ сработала
  ли ложно там, где не должна (`info`, `offtopic`, `edge_conflict`, `edge_no_data`).

Детальный отчёт по каждому кейсу — в `eval_report.json`.

## Наблюдения по качеству

На GigaChat-2 стабильно сильны: `info`, `transactional`, обе категории эскалации (триггеры
≈100%), `offtopic`. Краевые случаи (`edge_no_data`, `edge_conflict`, `edge_manipulation`)
дают разброс — модель иногда не делает отдельный поиск по граничным правилам (например,
порог сегмента «Средний бизнес» 800 млн руб., п. 1.2 РП-КР-001) и не всегда твёрдо отказывает.
Это ожидаемое поведение учебного кейса: датасет специально содержит коллизии и манипуляции,
а eval по категориям как раз и подсвечивает эти зоны для доработки промпта/ретрива.

## Возможные улучшения
- Локальные эмбеддинги (sentence-transformers) как фолбэк при лимитах GigaChat API.
- Отдельный «eligibility»-шаг: до рекомендации продукта проверять сегмент и требования.
- Few-shot примеры отказов в промпте для краевых категорий.
