# Агент поддержки кредитования МСБ (LangGraph + GigaChat)

автоматизированный Помощник для обращений клиентов МСБ по
кредитованию. Подключает **RAG** по нормативке, **инструменты** к БД клиентов, корректно
**эскалирует** (продажи/негатив/безопасность) и **отказывает** в краевых случаях.

## Архитектура

```
src/credit_agent/
  config.py    # токен из ./env, фабрики GigaChat LLM и Embeddings, пути, TODAY
  prompts.py   # системный промпт по регламенту РП-ОБ-005 + контекст сессии + промпт рефлексии
  rag.py       # чанкинг доков по заголовкам (с иерархией), GigaChat embeddings, FAISS
  tools.py     # read-only инструменты к clients.sqlite + расчёт ДП + escalate()
  memory.py    # долговременная память сессии (remember/recall, JSON в .memory/)
  agent.py     # сборка LangGraph create_react_agent + рефлексия + guard текстовых tool-call'ов
build_index.py # построение FAISS-индекса (запустить один раз)
chat.py        # интерактивный CLI-чат
eval.py        # E2E-оценка по qa.jsonl (+ --resume-errors для дозапуска упавших кейсов)
metrics_axes.py     # метрики по осям: difficulty / is_multiturn / документ + графики + before/after
analyze_failures.py # распричинка fail-mode'ов (типы провалов, кластеры) для слайдов
trace.py            # сбор полных трейсов (reasoning + tool/RAG-вызовы) для слайдов
slides/        # architecture.drawio / .dot / .png — диаграмма архитектуры
charts/        # PNG-графики метрик (генерит metrics_axes.py)
PRESENTATION.md     # план 5-минутной презентации
```

### Ключевые решения
- **RAG**: документы режутся по заголовкам markdown с сохранением *пути* родительских
  заголовков в каждом чанке (дочерняя «Условия» несёт название продукта) — это резко
  улучшает семантический поиск. Плюс синтетический обзорный чанк со списком всех 5 продуктов.
  Каждый чанк помнит `source` и номер пункта (`2.1.5`) для ссылок на нормативку.
- **Безопасность данных**: клиентские инструменты замкнуты на `client_id` сессии и НЕ
  принимают id от модели — доступ к данным третьих лиц закрыт на уровне кода (разд. 7 РП-ОБ-005).
- **Эскалация** — инструмент `escalate(trigger, summary)` с триггерами `sales|negative|security`;
  даёт машинно-проверяемый сигнал для оценки.
- **Идентификация**: анонимные каналы (`chat_site`/`contact_center`) → только общая инфа;
  авторизованные (`chat_intern`/`mobile`) → доступ к собственным договорам/заявкам.
- **Память** (`memory.py`): агент сам решает, что сохранить (цель обращения, интересующий
  продукт, желаемая сумма/срок, предпочтения) через инструмент `remember`, и поднимает
  контекст прошлых ходов/сессий через `recall`. Локальная долговременная память (JSON в
  `.memory/`, ключ — `client_id`).
- **Рефлексия** (`agent.run`): после черновика ответа выполняется самооценка всей цепочки
  относительно цели и ограничений регламента; при найденных недочётах агент один раз
  переигрывает ответ с подсказкой, снова имея доступ к инструментам. Fail-open: сбой
  самопроверки не ломает базовый ответ. Отключается флагом `CreditAgent(reflect=False)`.
- **Guard текстовых tool-call'ов** (`_recover_text_toolcall`): GigaChat-2 иногда печатает
  вызов инструмента ТЕКСТОМ (`escalate(...)`, `trigger="sales"`, `remember(...)`) вместо
  структурного tool-call — тогда реального действия не происходит. Guard детерминированно
  ловит это: для эскалации — поднимает реальный сигнал и заменяет ответ на корректный
  hand-off (не показывая клиенту «сырой» вызов); прочие текстовые вызовы трактуются как
  незавершённый ход и безопасно переигрываются. Не зависит от «послушности» модели.
- **Авторизация GigaChat** (`config.py`): предпочтительно `GIGACHAT_CREDENTIALS` (auth key) —
  клиент сам обновляет короткоживущий токен, поэтому длинные прогоны (eval/traces) не падают
  по «Token has expired». Фолбэк — статический `GIGACHAT_ACCESS_TOKEN` (живёт ~30 мин).

## Архитектура (поток одного хода)

```
                 ┌──────────────────────── CreditAgent (сессия: client_id, channel) ─────────────────────┐
   обращение ──► │  build_session_context ─► SYSTEM_PROMPT                                                │
   (+ history)   │                                                                                        │
                 │   ┌─ ReAct-граф (LangGraph create_react_agent, GigaChat-2) ──────────────────────┐     │
                 │   │  reasoning ⇄ tool-calls:                                                      │     │
                 │   │    • search_regulations ─► FAISS (.index) ◄─ GigaChat Embeddings ◄─ documents │     │
                 │   │    • get_client_profile / get_active_loans / get_applications ─► clients.sqlite│    │
                 │   │    • calc_early_repayment ─► clients.sqlite (read-only)                        │     │
                 │   │    • remember / recall ─► .memory/<client_id>.json                            │     │
                 │   │    • escalate(trigger, summary) ─► escalation_sink                            │     │
                 │   └──────────────────────────────────────────────────────────────────────────────┘     │
                 │            │ черновик ответа                                                            │
                 │            ▼                                                                            │
                 │   _reflect (self-check, GigaChat) ── недочёты? ──да──► переигровка хода (1 раз) ────────┘
                 │            │ нет                                                                        │
   ответ ◄───────┤            ▼ финальный ответ (+ escalations, + reflection)                             │
                 └────────────────────────────────────────────────────────────────────────────────────────┘
```

## Установка

```bash
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
```

Авторизация GigaChat читается из файла `./env` или из окружения. Предпочтительно
`GIGACHAT_CREDENTIALS=<auth key>` (авто-обновление токена; не падает на длинных прогонах);
фолбэк — `GIGACHAT_ACCESS_TOKEN=<token>` (живёт ~30 мин). Модель — `GigaChat-2`, `temperature=0.1`.

## Запуск

```bash
# 1. Построить индекс RAG (один раз; пересобрать при изменении документов)
.venv/bin/python build_index.py

# 2. Чат
.venv/bin/python chat.py                                   # анонимная сессия
.venv/bin/python chat.py --client C-000001 --channel chat_intern   # авторизованная

# 3. E2E-оценка
.venv/bin/python eval.py --limit-per-category 2            # быстрый срез по всем категориям
.venv/bin/python eval.py --limit-per-category 99           # фактически полный прогон qa.jsonl
.venv/bin/python eval.py --resume-errors                  # дозапустить кейсы, упавшие из-за токена

# 4. Артефакты для презентации
.venv/bin/python metrics_axes.py --report eval_report.json \
    --before eval_report_before.json --mid eval_report_promptfix.json   # таблицы + charts/*.png
.venv/bin/python analyze_failures.py --report eval_report.json          # failure_analysis.md
.venv/bin/python trace.py --ids Q-001,Q-061,Q-091,Q-117 --prefix traces # полные трейсы (нужен токен)
```

## Метрики оценки (eval.py)

Считаются **по категориям отдельно** (как рекомендует README кейса):
- **judge_pass** — LLM-судья (GigaChat) сверяет ответ с `expected_behavior` по сути.
- **escalation** — детерминированная проверка триггера: сработала ли эскалация там, где
  ожидается (`escalation_sales`→`sales`, `escalation_negative`→`negative`), и НЕ сработала
  ли ложно там, где не должна (`info`, `offtopic`, `edge_conflict`, `edge_no_data`).

Детальный отчёт по каждому кейсу — в `eval_report.json`.

## Наблюдения по качеству (180 кейсов qa.jsonl)

Замеры — три точки на одних кейсах: `eval_report_before.json` (baseline) → `eval_report_promptfix.json`
(промпт-фикс) → `eval_report.json` (+ guard). Сводки: `metrics_by_axis.md`, `failure_analysis.md`,
графики `charts/*.png`. Цель — показать «было → стало», а не максимизировать число.

**Главная находка.** LLM-судья нередко ставил «pass», но **детерминированный** escalation-сигнал
ловил баг: GigaChat-2 печатал вызов инструмента ТЕКСТОМ (`trigger="sales"`, `escalate(...)`,
`remember(...)`) вместо структурного tool-call (**11/180 = 6%** кейсов) → реального hand-off не
было. Фикс промпта + детерминированный guard подняли sales-эскалацию **17% → 39% → 44%**, negative
**88% → 56% → 69%**. Guard только ДОБАВЛЯет эскалации (никогда не убирает), поэтому его вклад
монотонно неотрицателен; колебания baseline→промпт — это **стохастичность tool-calling GigaChat**
между прогонами (тот же агент даёт 88%↔56% на negative).

**По осям** (180 кейсов, общий judge 64%): сложность монотонна (easy 73% → medium 64% → hard 58%),
многоходовые труднее одноходовых (53% vs 67%). Сильны `info` (82%), эскалации; просадка — `transactional`
(точность расчётов/цифр, 49%) и краевые `edge_*` (коллизии/манипуляции, твёрдость отказа).

> Замечание по воспроизводимости: статический access-токен живёт ~30 мин, и длинные прогоны на нём
> упираются в «Token has expired» (упавшие кейсы — инфраструктура, не агент; дозапускаются
> `eval.py --resume-errors`). Для стабильных прогонов используйте `GIGACHAT_CREDENTIALS`
> (авто-обновление токена).

## Соответствие рубрике зрелости агента

| Признак (вес) | Уровень | Чем обеспечен |
|---|---|---|
| Reasoning (×3) | 2 | ReAct: недетерминированные цепочки, перепланирование по результатам поиска/рефлексии |
| Autonomy (×3) | 3 | Не запрашивает подтверждений промежуточных действий |
| Action (×2) | 2 | Сам формирует структурированные tool-calls (RAG, БД, память, эскалация) |
| Memory (×2) | 1 | `remember`/`recall` — агент сам пишет/читает локальную долговременную память |
| Reflection (×2) | 2 | Самооценка результата всей цепочки + переигровка при недочётах (`_reflect`) |
| Domain knowledge (×1) | 1 | RAG по нормативке (FAISS + GigaChat Embeddings) |
| Role (×1) | 1 | Роль задана system_prompt'ом по регламенту РП-ОБ-005 |

## Ограничения и направления развития
- **Память — уровень 1 (локальная).** До уровня 2 нужен внешний долговременный стор
  («Цифровой двойник» клиента) вместо JSON в `.memory/`.
- **Расчёт ДП приближённый.** Проценты начисляются от оценки даты последнего платежа
  (`calc_early_repayment`), без точного графика; для прод-расчёта нужен полный график платежей.
- **Зависимость от GigaChat API.** Нет фолбэка на локальные эмбеддинги (sentence-transformers)
  при лимитах/недоступности API; индекс надо пересобирать вручную при изменении документов.
- **Рефлексия удваивает LLM-вызовы** (минимум +1 на ход, +ещё прогон при переигровке) —
  заметно на больших прогонах; отключается `reflect=False`.
- **Краевые категории** (`edge_*`) дают разброс: модель не всегда делает отдельный поиск по
  граничным правилам и не всегда твёрдо отказывает. Возможные доработки: отдельный
  «eligibility»-шаг (проверка сегмента/требований до рекомендации продукта) и few-shot
  примеры отказов в промпте.
