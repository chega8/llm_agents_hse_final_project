## Слайд 1 — Титул (15 сек)
**Агент поддержки кредитования МСБ.** Стек: **LangGraph (ReAct) + GigaChat-2 + FAISS RAG**.
- Одна фраза: «Помощник отвечает по нормативке, работает с данными клиента, эскалирует и отказывает — всё с проверяемыми сигналами».

## Слайд 2 — Архитектура одного хода (60 сек)
Картинка: `slides/architecture.png`.
- Вход (обращение + history + channel + client_id) → системный промпт по регламенту.
- **ReAct-граф**: reasoning ⇄ tool-calls (недетерминированная цепочка).
- 4 группы инструментов: **RAG** (search_regulations→FAISS), **клиентские** (профиль/кредиты/заявки/расчёт ДП→sqlite), **память** (remember/recall), **эскалация** (escalate→sink).
- Поверх — **рефлексия**: самопроверка черновика, безопасная переигровка 1 раз.

## Слайд 3 — Ключевые решения реализации (60 сек)
Здесь — то, чем мы отличаемся (4 буллета):
1. **RAG с иерархией заголовков.** Чанки несут *путь* родительских заголовков (дочерняя «Условия» помнит название продукта) + синтетический обзорный чанк по всем продуктам → резко лучше ретрив. Каждый чанк хранит source и номер пункта для ссылок.
2. **Безопасность данных в коде, не в промпте.** Клиентские инструменты замкнуты на `client_id` сессии и **не принимают id от модели** → доступ к чужим данным закрыт архитектурно (разд. 7 регламента).
3. **Машинно-проверяемая эскалация.** `escalate(trigger, summary)` с триггерами sales/negative/security даёт детерминированный сигнал для метрик (а не «угадывание по тексту»).
4. **Рефлексия + локальная долговременная память.** Самооценка ответа всей цепочки (fail-open) и `remember`/`recall` по client_id.

## Слайд 4 — Из чего состоит агент (45 сек)
Признаки системы списком (не таблица):
- **ReAct agent loop** — tool → результат → следующее действие, автономно (Reasoning/Autonomy/Action).
- **RAG по нормативке** — факты только из поиска FAISS, не из памяти модели (Domain).
- **Клиентские тулы** — профиль/кредиты/заявки/расчёт ДП, замкнуты на client_id (безопасность в коде).
- **Долговременная память** — remember/recall: цель, продукт, сумма/срок (Memory).
- **Self-reflection** — доп. слой валидации черновика + переигровка (Reflection).
- **Эскалация как действие** — escalate(...): реальный hand-off + детерминир. сигнал (Action).
- **Роль и ограничения** — system-промпт по регламенту (Role).

## Слайд 7 — Находка: что eval поймал, а судья пропустил (45 сек)
- **Симптом:** LLM-судья ставил «pass», но **детерминированный** escalation-сигнал показывал sales = **17%**.
- **Причина:** GigaChat-2 печатал вызов инструмента **ТЕКСТОМ** — `trigger="sales"`, `escalate(...)`, `remember(...)` — вместо структурного tool-call. Для пользователя похоже на ответ, но **реального действия (hand-off / запись в память) не происходит.**
- Масштаб: **11/180 кейсов (6%)**, стохастично — в разных прогонах попадает на разные кейсы.
- **Мораль:** одного LLM-судьи мало; детерминированный сигнал по факту вызова инструмента ловит то, что судья не видит.

## Слайд 8 — Что именно пофиксили (45 сек)
Два слоя, оба в репозитории:
1. **Промпт (`prompts.py`):** триггеры переписаны как действия + явное «эскалация выполняется ТОЛЬКО вызовом инструмента, не пиши `trigger=` текстом». Снижает частоту бага.
2. **Детерминированный guard (`agent.py`, `_recover_text_toolcall`):** пост-фактум ловит текстовый вызов в ответе → **поднимает реальную эскалацию** и заменяет ответ на корректный hand-off (клиент не видит «сырой» вызов). Не зависит от «послушности» модели; **только добавляет** эскалации, никогда не убирает.
- Бонус: `GIGACHAT_CREDENTIALS` (авто-обновление токена) — длинные прогоны eval не падают на «Token expired».

```
ответ модели:  trigger="sales" ...        ❌ текстовый «вызов», действия нет
                      │ guard
                      ▼
escalations += {sales}  +  ответ → «Передаю обращение специалисту…»   ✅
```

## Слайд 9 — Было → стало (45 сек)
Картинка: `charts/escalation_beforeafter.png`.
- Escalation (детерминир.): sales **17% → 39% → 44%**, negative **88% → 56% → 69%** (baseline→промпт→guard).
- Guard даёт монотонный плюс (mid→guard ≥ 0 всегда).
- **Честно:** negative baseline был 88% — tool-calling GigaChat **стохастичен** между прогонами (тот же агент: 88%↔56%); guard ограничивает просадку, но не перебивает дисперсию модели. Цель — не накрутить число, а показать механику фикса.
